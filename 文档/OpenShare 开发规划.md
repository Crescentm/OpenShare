# OpenShare 开发规划

## 技术栈落地说明

- 前端技术栈：Vue 3 + Vite + TypeScript + Tailwind CSS
- 后端技术栈：Go + Gin + GORM
- 数据库：PostgreSQL
- 文件存储：Linux Local File System
- 认证方式：JWT Token
- 密码加密：bcrypt
- 管理端请求头：Authorization: Bearer Token

开发计划默认以以上技术栈为前提推进，后续阶段中提到的实现方式均以该技术选型为基础。

## 阶段一：项目初始化与基础架构

### 1.1 后端项目初始化
- 创建 Go 项目结构（cmd、internal、pkg、configs、migrations 等目录）
- 配置 Gin 框架，设置路由分组
- 集成 GORM，配置 PostgreSQL 连接
- 配置文件管理（yaml/env 配置加载）
- 统一错误处理与响应格式
- 日志系统搭建
- 增加健康检查接口
- 建立 API 分层结构，区分 handler、service、repository、model
- 约定 Gin 中间件链，统一处理日志、恢复、鉴权、限流、异常响应

### 1.2 前端项目初始化
- 使用 Vite 创建 Vue 3 + TypeScript 项目
- 配置 Tailwind CSS
- 配置路由（Vue Router）
- 配置状态管理（Pinia）
- 封装 Axios 请求模块
- 配置全局样式与基础布局
- 配置环境变量区分开发与生产环境
- 统一封装请求拦截器和响应拦截器，处理 Token 注入和错误提示
- 划分用户端与管理端路由模块

### 1.3 开发环境初始化
- 准备 PostgreSQL 本地开发环境
- 约定项目运行端口、接口前缀、跨域策略
- 初始化 Git 忽略规则和基础脚本
- 约定目录规范、命名规范、接口返回规范
- 建立基础 README 和启动说明
- 约定 PostgreSQL 扩展启用策略，为后续 pg_trgm 和全文搜索做准备

### 1.4 存储目录初始化
- 创建本地文件存储目录
- 建立 repository、staging、trash 三个核心目录
- 启动时检测目录是否存在，不存在则自动创建
- 校验目录读写权限
- 默认按 Linux 本地文件系统设计路径规则，避免把存储逻辑写死在业务层

## 阶段二：数据库设计与模型层建设

### 2.1 核心数据表设计
- 设计 admins 表，存储管理员账号和权限信息
- 设计 files 表，存储文件元数据
- 设计 folders 表，存储文件夹结构
- 设计 submissions 表，存储上传审核记录
- 设计 tags 表，存储标签信息
- 设计 file_tags、folder_tags 关联表
- 设计 reports 表，存储举报记录
- 设计 announcements 表，存储公告内容
- 设计 operation_logs 表，存储操作审计日志

### 2.2 模型与迁移实现
- 编写 GORM 模型
- 建立数据库迁移脚本
- 初始化基础索引和唯一约束
- 为文件名冲突、Tag 重名等规则增加数据库层约束
- 对常用筛选字段建立普通索引，如状态、创建时间、父目录、回执码
- 为后续搜索字段预留 tsvector 或等价索引策略

### 2.3 状态流转设计
- 定义 submission 状态：pending、approved、rejected
- 定义资源状态：active、offline、deleted
- 明确审核通过、驳回、下架、删除时的状态变化规则
- 明确数据库状态与磁盘文件状态的同步规则

### 2.4 文件元数据规范
- 统一文件 UUID 作为主标识
- 保存原始文件名、文件大小、类型、路径、上传时间
- 记录文件所属目录、关联 Tag、下载次数
- 预留预览能力所需字段
- 明确数据库保存元数据，磁盘保存真实文件内容，避免把二进制文件写入数据库

## 阶段三：认证与权限系统

### 3.1 超级管理员初始化
- 系统首次启动时自动生成 Super Admin
- 随机生成初始密码
- 将初始密码以安全方式输出
- 避免重复初始化超级管理员账号

### 3.2 管理员登录能力
- 实现管理员登录接口
- 使用 bcrypt 存储密码哈希
- 使用 JWT 签发访问令牌
- 配置登录失败处理和错误提示
- 统一要求管理端通过 Authorization: Bearer Token 传递访问令牌
- 明确 Token 过期策略和失效后的前端跳转逻辑

### 3.3 权限中间件
- 建立认证中间件，校验 JWT
- 建立角色中间件，区分 Guest、Admin、Super Admin
- 建立权限项校验中间件
- 支持普通管理员的可配置权限
- 将登录校验与具体权限校验拆分，避免把所有逻辑耦合在单一中间件中

### 3.4 管理员权限模型
- 设计权限项，如审核资料、发布公告、删除资料、管理 Tag
- 支持超级管理员为普通管理员分配权限
- 预留后续扩展权限项的结构

---

## 阶段四：资料核心链路开发

### 4.1 资料浏览功能
- 实现公开资料列表接口
- 返回标题、Tag、上传时间、下载量、文件大小
- 支持分页、排序、目录浏览
- 隐藏上传者信息

### 4.2 资料上传功能
- 实现游客上传接口
- 支持 multipart/form-data 上传
- 上传文件先写入 staging 目录
- 录入 submission 记录
- 支持标题、描述、Tag、回执码填写
- 支持用户自定义回执码或系统自动生成回执码
- 基于 Gin FormFile 和 SaveUploadedFile 实现上传落盘
- 上传完成后采用“写入 staging -> 创建 submission 记录”的顺序保证流程清晰

### 4.3 投稿记录查询
- 实现根据回执码查询投稿记录接口
- 返回标题、审核状态、上传时间、下载量
- 支持浏览器本地缓存回执码
- 返回状态值时统一使用 pending、approved、rejected，避免前后端枚举不一致

### 4.4 管理员审核功能
- 实现待审核列表接口
- 实现审核通过接口
- 实现审核驳回接口，并记录原因
- 审核通过后将文件从 staging 移动到 repository
- 审核驳回后保留记录并标记状态
- 审核通过时使用 rename 完成 staging 到 repository 的文件移动
- 审核动作需要同时更新 submission 状态和资源公开状态

### 4.5 资料下载功能
- 实现单文件下载接口
- 下载时记录下载次数
- 设置文件下载响应头
- 对不存在或未公开文件返回正确状态码
- 单文件下载优先使用 Gin c.FileAttachment()
- 下载计数更新与文件响应建议解耦，避免计数失败影响主下载链路

### 4.6 资料发布规则
- 仅 approved 状态的资源允许公开展示
- 未审核或已驳回资源不能出现在公开列表
- 后台可查看全部审核记录和状态

## 阶段五：搜索系统开发

### 5.1 搜索能力建设
- 启用 PostgreSQL Full Text Search
- 启用 pg_trgm 扩展
- 为标题、描述、Tag、目录名称建立搜索索引
- 明确全文搜索负责关键词相关度匹配，pg_trgm 负责模糊匹配与近似搜索
- 统一搜索字段清洗规则，降低大小写和空白差异带来的匹配偏差

### 5.2 搜索接口开发
- 支持关键词搜索
- 支持模糊匹配
- 支持 Tag 过滤
- 支持 Tag 组合搜索
- 支持限定目录范围搜索
- 支持大小写不敏感搜索
- 支持文件和文件夹同时参与搜索
- 将文件继承父文件夹 Tag 的逻辑纳入搜索条件构建
- 为后续搜索建议保留 query 参数、tag 参数、folder 参数等结构化接口形式

### 5.3 搜索排序与优化
- 按相关度排序
- 按下载量辅助排序
- 处理空结果和低匹配度结果
- 针对常用搜索字段做索引优化
- 优先使用 PostgreSQL 排序能力完成相关度与下载量组合排序
- 避免先全量取出结果再在应用层排序

## 阶段十：文件预览与批量下载

### 10.1 文件预览功能
- 支持 PDF 在线预览
- 支持图片在线预览
- 支持纯文本在线预览
- 对不支持预览的格式统一提示下载
- 不引入 Office 在线转换链路，保持系统轻量和稳定

### 10.2 批量下载功能
- 支持多文件勾选下载
- 支持打包下载
- 使用 Streaming ZIP 进行流式压缩输出
- 控制大批量下载时的资源占用
- 基于 archive/zip 和 io.Copy 实现流式压缩下载
- 避免先生成完整压缩包再落盘，减少磁盘和内存占用

### 10.3 文件详情页能力
- 展示文件基础信息
- 展示 Tag、上传时间、下载次数、文件大小
- 根据文件类型决定预览或下载模式

## 阶段十一：安全与审计能力

### 11.1 操作日志
- 记录上传者 IP
- 记录管理员操作 IP
- 记录审核、删除、下架、公告发布、权限调整等关键操作
- 提供后台日志查询能力
- 日志字段建议至少包含操作人、角色、动作、目标对象、IP、时间、结果

### 11.2 防刷机制
- 对上传接口进行限流
- 对搜索接口进行限流
- 必要时对下载接口做基础限流
- 记录异常访问行为
- 限流优先按 IP 和接口维度实施，符合当前免登录场景

### 11.3 数据一致性保障
- 审核通过时使用事务控制数据库更新
- 文件移动失败时增加补偿机制
- 删除和下架操作优先采用软删除或移入 trash
- 增加异常场景日志记录
- 明确处理“数据库成功但文件失败”与“文件成功但数据库失败”两类异常路径
