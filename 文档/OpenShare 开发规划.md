# OpenShare 开发规划

## 1. 文档目的

本文档用于指导 OpenShare 按阶段逐步开发。目标不是只列出功能，而是把每一阶段拆成可以直接执行的步骤、输出物、依赖关系和验收标准，后续开发可以严格按本文档推进。

## 2. 当前技术基线

开发规划默认以当前技术文档为准：

- 前端：Vue 3 + Vite + TypeScript + Tailwind CSS
- 后端：Go + Gin + GORM
- 数据库：SQLite
- 文件存储：Linux Local File System
- 管理端认证：Session（Cookie）
- 密码加密：bcrypt

## 3. 整体开发策略

### 3.1 总体原则

- 先做能跑通的主链路，再补治理、体验和运维能力
- 先固定数据模型和状态流转，再写接口和页面
- 文件系统与数据库一致性优先于界面体验
- 每个阶段必须有可演示结果，避免长周期堆积
- 每个阶段结束时都要同步更新文档，不把设计留在脑中

### 3.2 推荐开发顺序

建议按以下顺序推进，而不是并行平均铺开：

1. 项目初始化和基础设施
2. 数据库模型与迁移
3. Session 认证与管理员权限
4. 上传、审核、浏览、下载主链路
5. 搜索与 Tag
6. 后台管理能力
7. 举报、审计、限流
8. 预览、批量下载、本地导入
9. 联调、测试、部署

### 3.3 每个阶段固定产出

每一阶段结束时必须至少交付：

- 对应代码
- 对应配置和迁移
- 接口清单
- 页面或命令行可验证入口
- 自测结果
- 已知限制

## 4. 全局设计约束

### 4.1 后端架构约束

- 目录建议：`cmd`、`internal`、`pkg`、`configs`、`migrations`
- 分层建议：`handler -> service -> repository -> model`
- GORM 负责模型映射、查询和事务，不要把复杂业务逻辑堆进 model hook
- 文件操作必须单独封装到 storage 模块，避免在业务代码里散落路径拼接
- 所有状态流转必须由 service 层控制，不要在 handler 层直接改数据库

### 4.2 前端架构约束

- 用户端和管理端路由分离
- 请求层统一封装，统一处理错误、登录失效和提示
- 管理端依赖 Cookie Session，前端请求必须正确处理凭证透传
- 列表页统一分页、排序、筛选和空状态
- 状态值直接以后端枚举为准，不在前端自行定义另一套逻辑状态

### 4.3 数据一致性约束

- 数据库存元数据，磁盘存文件内容
- `submission` 审核状态与资源公开状态分离
- 审核通过时必须同时处理数据库更新和文件移动
- 删除优先走软删除或移入 `trash`
- 必须明确处理以下异常：
  - 数据库成功，文件失败
  - 文件成功，数据库失败
  - 文件丢失，数据库仍显示可下载

### 4.4 SQLite 与搜索约束

- 搜索统一采用 SQLite FTS5
- FTS5 索引字段以技术文档为准：文件名、文件夹名、Tag
- 描述字段是否进入全文索引，后续如果要做，必须先更新技术文档再实现
- 普通筛选条件和排序字段仍走业务表索引，不要全部依赖 FTS5
- Tag 重名校验必须忽略大小写

### 4.5 认证与安全约束

- 后台认证采用 Session（Cookie），不是 JWT
- Session 需要明确有效期、续期策略、登出销毁策略
- Cookie 需要明确 `HttpOnly`、`SameSite`、`Secure` 策略
- 上传和搜索接口要有限流能力
- 需要记录上传者 IP 和管理员操作 IP

## 6. 详细开发计划

## 阶段一：项目初始化与基础架构

### 阶段目标

完成前后端基础骨架、运行方式、配置加载和目录初始化，为后续编码提供稳定地基。

### 开发步骤

#### 1.1 建立仓库结构

- 创建后端目录：`cmd/server`、`internal/config`、`internal/router`、`internal/handler`、`internal/service`、`internal/repository`、`internal/model`、`internal/storage`、`internal/middleware`、`pkg`、`configs`、`migrations`
- 创建前端目录并初始化 Vite 工程
- 建立文档目录规范，后续接口文档、部署文档、自测记录都放到固定位置

#### 1.2 后端基础启动能力

- 初始化 Go Module
- 集成 Gin
- 增加健康检查接口
- 预留统一路由注册入口
- 建立基础日志输出

#### 1.3 配置管理

- 定义配置文件结构：服务端口、数据库路径、存储根目录、Session 配置、限流配置
- 约定配置读取顺序：默认配置 -> 本地配置 -> 环境变量覆盖
- 增加启动时配置校验，避免配置缺失后运行到一半报错

#### 1.4 数据库连接初始化

- 接入 SQLite
- 接入 GORM
- 配置连接初始化、日志级别、自动基础校验
- 不依赖 AutoMigrate 作为正式迁移方案，迁移仍以显式 SQL 或明确脚本为主

#### 1.5 文件存储目录初始化

- 定义 `/data/openshare` 下的 `repository`、`staging`、`trash`
- 增加启动时目录存在性检查
- 增加读写权限检查
- 存储目录路径统一由配置项控制

#### 1.6 前端基础能力

- 初始化 Vue 3 + TypeScript + Tailwind CSS
- 接入 Vue Router、Pinia
- 建立用户端和管理端两个布局
- 封装 HTTP 请求模块
- 开启带 Cookie 的请求配置，保证 Session 能正常使用

### 本阶段输出物

- 前后端项目骨架
- 配置文件模板
- 健康检查接口
- 统一目录规范
- 启动说明文档

### 本阶段验收标准

- 前端可以启动并看到基础页面
- 后端可以启动并访问健康检查接口
- SQLite 数据库可以正常连接
- 缺失存储目录时服务可以自动创建
- 配置错误时服务可以在启动阶段直接失败并给出明确信息

### 本阶段开发顺序建议

1. 后端骨架
2. 配置加载
3. SQLite + GORM 连接
4. 存储目录初始化
5. 前端骨架
6. README 和启动方式

## 阶段二：数据库设计与模型层建设

### 阶段目标

把核心业务表、状态、索引和迁移策略定下来，后续功能都围绕这套结构扩展。

### 开发步骤

#### 2.1 明确核心实体

必须先定义清楚以下实体及关系：

- 管理员 `admins`
- 文件 `files`
- 文件夹 `folders`
- 上传审核记录 `submissions`
- 标签 `tags`
- 文件标签关系 `file_tags`
- 文件夹标签关系 `folder_tags`
- 举报记录 `reports`
- 公告 `announcements`
- 操作日志 `operation_logs`
- 候选 Tag 提交 `tag_submissions`（如果保留该功能）

#### 2.2 明确每张表的关键字段

至少在文档里先定稿以下字段：

- 主键
- 创建时间
- 更新时间
- 状态字段
- 关联对象字段
- 业务唯一键
- 软删除字段是否需要

#### 2.3 明确状态流转

- `submission`：`pending`、`approved`、`rejected`
- `resource`：`active`、`offline`、`deleted`
- `report`：`pending`、`approved`、`rejected`

每一种状态变化都要明确定义：

- 谁触发
- 什么时候触发
- 修改哪些字段
- 是否需要记录原因
- 是否需要写操作日志

#### 2.4 设计文件相关字段

文件表建议至少包含：

- `id`
- `folder_id`
- `submission_id`
- `title`
- `original_name`
- `stored_name`
- `extension`
- `mime_type`
- `size`
- `disk_path`
- `status`
- `download_count`
- `uploader_ip`
- `created_at`
- `updated_at`

其中：

- `title` 用于展示
- `original_name` 用于保留原始文件名
- `stored_name` 或路径字段用于磁盘定位

#### 2.5 设计 submission 字段

建议至少包含：

- `id`
- `receipt_code`
- `title_snapshot`
- `description_snapshot`
- `status`
- `reject_reason`
- `uploader_ip`
- `reviewer_id`
- `reviewed_at`
- `created_at`
- `updated_at`

#### 2.6 索引与唯一约束

- 回执码唯一索引
- Tag 名称忽略大小写唯一
- 同目录文件名冲突约束
- 常见列表条件建立索引：状态、创建时间、父目录、审核时间
- 搜索使用 FTS5 虚表

#### 2.7 迁移方案

- 编写初始化迁移脚本
- 编写后续增量迁移规范
- 明确开发环境如何重建数据库
- 明确生产环境如何安全升级

### 本阶段输出物

- 表结构文档
- 状态流转文档
- 首批迁移脚本
- GORM 模型定义

### 本阶段验收标准

- 新数据库可一键初始化
- 关键索引和唯一约束生效
- 表结构足以支撑上传、审核、下载、搜索、举报、公告
- 后续阶段不需要再大改主表结构

### 本阶段开发顺序建议

1. 先画实体关系
2. 再定字段
3. 再定状态流转
4. 再写迁移
5. 最后写 GORM 模型

## 阶段三：认证与权限系统

### 阶段目标

完成管理端登录、Session 维持和权限控制，为审核后台提供入口。

### 开发步骤

#### 3.1 Super Admin 初始化

- 系统首次启动时检查是否已有超级管理员
- 若不存在则创建默认超级管理员
- 随机生成初始密码
- 初始密码只输出一次
- 避免重复初始化

#### 3.2 Session 方案落地

- 选定 Session 存储方式
- 当前阶段可先采用 SQLite 持久化或内存 Session，若选内存需在文档里明确限制
- 配置 Cookie 名称、有效期、续期方式、Path、SameSite、HttpOnly
- 生产环境预留 Secure 配置

#### 3.3 管理员登录接口

- 账号密码登录
- bcrypt 比对
- 登录成功后写入 Session
- 返回管理员基础信息

#### 3.4 管理员登出接口

- 销毁 Session
- 清理 Cookie

#### 3.5 认证中间件

- 判断 Session 是否存在
- 从 Session 中恢复管理员身份
- 注入请求上下文

#### 3.6 权限中间件

- 按角色和权限项校验
- 支持 Super Admin 全权限
- 支持普通管理员细粒度授权

#### 3.7 管理员模型与权限模型

- 定义管理员状态：启用、停用
- 定义权限项：审核、公告、资源编辑、资源删除、Tag 管理、管理员管理、系统设置

### 本阶段输出物

- 登录接口
- 登出接口
- Session 中间件
- 权限中间件
- 超级管理员初始化说明

### 本阶段验收标准

- 管理员可以登录后台
- 浏览器刷新后 Session 仍有效
- 未登录访问后台接口会被拦截
- 无权限管理员访问敏感接口会返回 403

### 本阶段开发顺序建议

1. 超级管理员初始化
2. 登录接口
3. Session 中间件
4. 登出接口
5. 权限模型
6. 权限中间件

## 阶段四：资料核心链路开发

### 阶段目标

完成系统最关键的主流程：上传、回执码查询、管理员审核、公开浏览、下载。

### 开发步骤

#### 4.1 公开资料列表

- 实现公开资料列表接口
- 只返回公开资源
- 返回字段：标题、Tag、上传时间、下载量、文件大小
- 支持分页、排序、目录浏览
- 不返回上传者信息

#### 4.2 游客上传接口

- 支持 `multipart/form-data`
- 接收标题、描述、Tag、文件、回执码
- 支持用户自定义回执码
- 如果未提供回执码则系统自动生成
- 基础校验：空文件、超大文件、非法扩展名、非法 MIME、标题缺失

#### 4.3 上传落盘流程

流程固定为：

1. 接收上传文件
2. 写入 `staging`
3. 创建 `submission`
4. 创建初始文件元数据或待审核资源记录
5. 返回回执码

必须明确：

- 文件名冲突如何处理
- 上传过程中断如何清理临时文件
- 回执码生成失败如何处理

#### 4.4 回执码查询

- 根据回执码查询投稿记录
- 返回审核状态、上传时间、下载量、驳回原因
- 浏览器缓存最近回执码

#### 4.5 审核后台接口

- 待审核列表
- 审核通过
- 审核驳回
- 支持记录驳回原因

#### 4.6 审核通过流程

必须固定执行顺序：

1. 校验 submission 状态必须是 `pending`
2. 校验 staging 文件存在
3. 将文件移动到 `repository`
4. 更新文件资源状态为公开可用
5. 更新 submission 状态为 `approved`
6. 记录审核人、审核时间、操作日志

#### 4.7 审核驳回流程

建议流程：

1. 校验 submission 状态必须是 `pending`
2. 更新 submission 状态为 `rejected`
3. 写入驳回原因
4. 资源保持未公开
5. 记录操作日志

是否立即删除 staging 文件，需要在正式编码前定稿。

#### 4.8 单文件下载

- 根据资源状态校验是否可下载
- 使用 `http.ServeContent`
- 正确设置文件响应头
- 下载计数更新与主下载链路解耦

### 本阶段输出物

- 上传接口
- 回执码查询接口
- 审核接口
- 公开列表接口
- 单文件下载接口
- 前台上传页和管理台审核页最小版本

### 本阶段验收标准

- 用户能上传一个真实文件并拿到回执码
- 管理员能在后台看到待审核记录
- 审核通过后文件可被浏览和下载
- 审核驳回后文件不出现在公开列表
- 文件移动失败时不会产生伪公开数据

### 本阶段开发顺序建议

1. 公开资料列表
2. 上传接口
3. 回执码查询
4. 审核列表
5. 审核通过/驳回
6. 下载接口
7. 前后台最小页面

## 阶段五：搜索系统开发

### 阶段目标

让用户可以按关键词、Tag 和目录范围高效搜索文件和文件夹。

### 开发步骤

#### 5.1 FTS5 索引表设计

- 建立 FTS5 虚表
- 索引字段按技术文档实现：文件名、文件夹名、Tag
- 设计业务表与 FTS5 索引同步机制

#### 5.2 搜索输入清洗

- 去除首尾空格
- 统一大小写策略
- 处理多空格和特殊字符
- 防止非法查询导致 SQL 拼装风险

#### 5.3 搜索接口

- 关键词搜索
- 模糊匹配
- Tag 限定搜索
- Tag 组合搜索
- 限定目录范围搜索
- 同时搜索文件和文件夹

#### 5.4 Tag 继承搜索

- 文件继承父文件夹 Tag
- 查询时必须把继承 Tag 纳入条件
- 需要先确定继承 Tag 的实现方式：
  - 查询时动态拼接
  - 写入缓存关系表

#### 5.5 搜索排序

- 相关度排序
- 下载量辅助排序
- 空结果处理
- 低匹配度结果处理

### 本阶段输出物

- FTS5 设计说明
- 搜索接口
- 搜索页

### 本阶段验收标准

- 文件和文件夹都能被搜索到
- Tag 过滤结果正确
- 目录范围限制正确
- 搜索结果排序基本符合预期

### 本阶段开发顺序建议

1. FTS5 索引结构
2. 索引同步
3. 搜索接口
4. Tag 条件整合
5. 搜索页

## 阶段六：Tag 系统开发

### 阶段目标

把 Tag 从普通字段升级为独立治理系统。

### 开发步骤

#### 6.1 Tag 基础管理

- Tag 创建
- Tag 编辑
- Tag 删除
- Tag 列表查询
- 忽略大小写重名校验

#### 6.2 文件和文件夹绑定 Tag

- 文件绑定 Tag
- 文件夹绑定 Tag
- 修改绑定关系时同步搜索索引

#### 6.3 文件继承父文件夹 Tag

- 明确继承范围
- 明确展示逻辑
- 明确搜索逻辑
- 明确后台编辑时是展示直接 Tag、继承 Tag 还是两者都展示

#### 6.4 用户候选 Tag

- 用户提交候选 Tag
- 管理员审核候选 Tag
- 审核通过后写入正式 Tag 表

#### 6.5 Tag 治理

- 合并 Tag
- 迁移关联关系
- 删除后的关联清理

### 本阶段输出物

- Tag 接口
- Tag 管理页面
- Tag 继承规则说明

### 本阶段验收标准

- 不会出现大小写不同但语义相同的重复 Tag
- 文件、文件夹、搜索三处 Tag 表现一致

## 阶段七：举报系统开发

### 阶段目标

补足内容治理闭环。

## 7. 建议先定稿的关键细节

这些问题建议在正式写代码前先确定，否则一定返工：

1. GORM 模型字段最终版，尤其是 `files`、`submissions`、`reports`
2. Session 存储方式、Cookie 策略、有效期与续期策略
3. 回执码生成规则、长度、唯一性、是否区分大小写
4. 审核通过时数据库事务和文件移动的边界
5. 驳回后 staging 文件是否保留、保留多久、何时清理
6. Tag 继承是查询时计算还是写入缓存关系
7. FTS5 索引同步策略
8. 批量下载的数量、总大小和超时上限
9. 导入本地目录时如何识别“同一个文件已导入”

## 8. 最推荐的实际开发顺序

如果后续准备一步步开发，建议严格按下面顺序进行：

1. 阶段一全部完成
2. 阶段二先完成表结构文档，再写迁移和 GORM 模型
3. 阶段三只做管理员登录和 Session，不要先做复杂权限页
4. 阶段四完整打通主链路并做最小页面
5. 主链路稳定后再上搜索和 Tag
6. 后台运营能力、举报、审计在主链路稳定后继续补
7. 最后做预览、批量下载、导入、测试和部署

## 9. 开发时的执行纪律

- 每完成一个阶段就更新开发规划中的完成状态
- 每增加一个状态字段就同步更新状态流转说明
- 每增加一个接口就同步补接口文档
- 每修改搜索字段或 Tag 规则，就同步检查 FTS5 和继承逻辑
- 每改动文件流转逻辑，都要重新回归上传、审核、下载链路
