# OpenShare 索引与唯一约束设计

本文档对应阶段二 `2.6`，用于在迁移脚本编写前先定稿索引、唯一约束和 FTS5 搜索边界。

## 设计原则

- 唯一约束优先保证业务正确性，不为“方便恢复”牺牲一致性
- 普通列表查询走业务表索引，全文搜索走 FTS5，二者职责分离
- 软删除表的唯一约束必须考虑 `deleted_at` 或等效活动态字段，避免历史垃圾数据阻塞正常重建
- SQLite 能力范围内优先用明确索引和检查约束，不依赖应用层“自觉保证”

## 1. 唯一约束

### `submissions`

- `receipt_code` 唯一索引

说明：

- 用于游客查询投稿记录，必须全局唯一
- 建议约束名称：`ux_submissions_receipt_code`

### `tags`

- `name_normalized` 唯一索引

说明：

- `name_normalized` 保存标准化后的标签名，例如统一转小写并裁剪首尾空白
- 通过该字段实现“Tag 名称忽略大小写唯一”
- 建议约束名称：`ux_tags_name_normalized`

### `files`

- 同目录文件名冲突约束

推荐方案：

- 约束维度：`(folder_id, stored_name, deleted_at)`

说明：

- 用 `stored_name` 作为磁盘定位名，直接承接技术文档里的“同目录文件名冲突返回 409”
- 对根目录文件，需要明确 `folder_id` 为空时的唯一性实现，迁移阶段建议配合表达式索引或约定根目录哨兵值
- 如果 SQLite 对 `NULL` 处理导致唯一性语义不够直观，迁移阶段应改用 `parent_folder_key` 一类非空规范化字段

### `folders`

- 同父目录目录名冲突约束

推荐方案：

- 约束维度：`(parent_id, name, deleted_at)`

说明：

- 与 `files` 的同目录唯一性策略保持一致
- 后续若需要忽略大小写，可增加 `name_normalized`，但当前文档尚未强制要求目录名忽略大小写唯一

### 关联表

- `file_tags(file_id, tag_id)` 唯一索引
- `folder_tags(folder_id, tag_id)` 唯一索引

说明：

- 防止重复绑定同一标签

## 2. 常见列表索引

### `submissions`

建议索引：

- `idx_submissions_status_created_at`：`(status, created_at DESC)`
- `idx_submissions_reviewer_id_reviewed_at`：`(reviewer_id, reviewed_at DESC)`
- `idx_submissions_reviewed_at`：`(reviewed_at DESC)`

用途：

- 后台待审核列表
- 已审核列表
- 按审核时间回溯

### `files`

建议索引：

- `idx_files_status_created_at`：`(status, created_at DESC)`
- `idx_files_folder_id_status`：`(folder_id, status)`
- `idx_files_submission_id`：`(submission_id)`

用途：

- 前台公开资源列表
- 目录下资源列表
- 通过 `submission` 反查生成资源

### `folders`

建议索引：

- `idx_folders_parent_id_status`：`(parent_id, status)`
- `idx_folders_status_created_at`：`(status, created_at DESC)`

用途：

- 目录树加载
- 按父目录浏览

### `reports`

建议索引：

- `idx_reports_status_created_at`：`(status, created_at DESC)`
- `idx_reports_reviewer_id_reviewed_at`：`(reviewer_id, reviewed_at DESC)`
- `idx_reports_file_id`：`(file_id)`
- `idx_reports_folder_id`：`(folder_id)`

用途：

- 待处理举报列表
- 已处理举报列表
- 资源举报历史查询

### `announcements`

建议索引：

- `idx_announcements_status_published_at`：`(status, published_at DESC)`

用途：

- 首页公告列表
- 后台公告管理列表

### `operation_logs`

建议索引：

- `idx_operation_logs_admin_id_created_at`：`(admin_id, created_at DESC)`
- `idx_operation_logs_target_type_target_id_created_at`：`(target_type, target_id, created_at DESC)`

用途：

- 管理员操作追溯
- 单资源审计链追溯

## 3. 检查约束建议

### `reports`

- `file_id` 与 `folder_id` 必须二选一

建议表达：

- `(file_id IS NOT NULL AND folder_id IS NULL) OR (file_id IS NULL AND folder_id IS NOT NULL)`

### `submissions`

- `reject_reason` 仅在 `status = rejected` 时必填
- `reviewer_id`、`reviewed_at` 在审核完成后必须同时存在

### `files` / `folders`

- `status = deleted` 时 `deleted_at` 必须非空

说明：

- 这类约束优先在迁移 SQL 中显式落地

## 4. FTS5 搜索设计

### 搜索对象

- `files`
- `folders`

### FTS5 索引字段

严格按技术文档定稿：

- 文件名
- 文件夹名
- Tag

不纳入当前 FTS5 范围：

- 描述字段
- 公告正文
- 举报原因

### 推荐虚表拆分

推荐建立一个独立虚表，例如 `search_index`，而不是分别为每张业务表建一套搜索表。

建议字段：

- `row_id`：对应业务对象主键
- `entity_type`：`file` 或 `folder`
- `name`
- `tags`

说明：

- 统一虚表更利于跨文件和目录混合搜索
- 实际搜索排序仍需结合业务表字段，例如下载量、状态、创建时间

### 数据同步策略

推荐通过触发器维护 FTS5：

- 新增文件/文件夹时写入
- 修改文件名/文件夹名时更新
- 标签绑定变更时更新
- 资源删除或软删除时移除或标记不可检索

说明：

- 查询时必须与业务表联查，过滤 `status != active` 或 `deleted_at IS NOT NULL` 的资源
- FTS5 只负责召回，不负责最终权限和状态判定

## 5. 本步骤结论

已定稿：

- 回执码唯一索引
- Tag 名称忽略大小写唯一
- 同目录文件名冲突约束
- 常见列表条件索引方向
- 搜索使用 FTS5 虚表

后续进入迁移阶段时还需最终确认：

- 根目录 `NULL folder_id` 的唯一性实现方式
- `deleted_at` 参与唯一约束时的 SQLite 兼容写法
- FTS5 触发器的具体 SQL 细节
